# 변경 원칙

## Scope

- 요청한 작업에 필요한 코드만 수정한다. 리팩터, 스타일 통일, "이것도 같이"는 하지 않는다.
- 이번 작업과 무관한 파일은 열지 않고, 건드리지 않는다.
- 테스트 파일(`*.test.ts`, `*.spec.ts`, `*.test.tsx`)은 요청이 있을 때만 추가·수정한다.

## Match existing code

- 수정 전에 같은 파일·같은 폴더의 패턴을 읽는다. 네이밍, import 순서, export 방식, 훅/서비스 사용 방식을 그대로 맞춘다.
- 이미 있는 함수·컴포넌트를 우선 재사용한다. 비슷한 로직을 새로 만들지 않는다.
- 타입도 마찬가지다. 새 타입을 정의하기 전에 `@frameon/types` 또는 같은 파일·폴더에 이미 있는지 확인한다.

## Diff quality

- 작고 읽기 쉬운 diff를 목표로 한다.
- 불필요한 주석, 장황한 docstring, 요청과 무관한 try/catch는 넣지 않는다.
- 다른 사람이 쓴 주석·코드는 요청이 없으면 삭제하지 않는다.
- 공백·들여쓰기·빈 줄 등 포맷 변경만 있는 diff는 만들지 않는다.

## Docs and assets

- 요청하지 않은 README, 설계 문서, 마크다운 가이드는 추가하지 않는다.
- 예시 파일, 더미 데이터, 스토리북 스토리도 요청이 있을 때만 추가한다.
